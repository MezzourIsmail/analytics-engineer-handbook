The patterns in this field guide didn't come from documentation. They came from six years of watching the same problems appear in different companies, scaling from solo engineer to running platform standards for 100+ engineers, and figuring out what actually works under production pressure. Alice's confusion, Thomas's shortcuts, Ismail's support queue, those stories are composites of situations I've been in or caused myself. The architecture patterns, the patterns, the tests, the configurations, each one traces back to a real decision made in a real production environment, usually because an earlier decision turned out to be wrong.

The common thread

Every company I worked in had a version of the same underlying problem. Some had too many tools and no shared standards. Some had standards that nobody followed because they lived in a wiki nobody read. Some had great engineers spending 30% of their time thinking about orchestration instead of building. Some had platform teams so buried in support tickets that they had stopped shipping anything new.

That experience is what shaped this guide. The materialization choices, the userland pattern and the query-tag macro in this guide all come from production, not documentation, each one a version of something I built to answer a real question like "which team burned 200 credits last Tuesday?" This is a field guide because it reflects how the field actually behaves, not how the documentation describes it.

The journey
February 2023 to now
Cloud Data Platform Engineer · Accor

The biggest platform challenge I had ever faced. More than 100 engineers across 15 distributed teams, all working on the same Snowflake warehouse but each doing it differently. My job was to standardise everything and make the standard so clear, so well-supported and so easy to follow that people like Alice stop getting lost, people like Thomas stop cutting corners, and the platform team stops spending its days answering the same questions on repeat.

dbt standardisation
  • Drove dbt adoption across 80+ data engineers
  • Built a scoring system across 6 dimensions: structure, naming, ownership, tests, docs, tags
  • GitLab CI/CD: standards enforced at merge, non-compliant code blocked automatically
  • Pre-commit hooks and sqlfmt for consistent SQL formatting
Airflow on Astronomer
  • Replaced 5 different orchestrators with one Airflow instance
  • Built reusable DAG templates with Cosmos
  • Deployment time cut from 5 minutes to 1 minute
  • 30% cost reduction from decommissioning legacy orchestrators
Snowflake governance
  • Full infrastructure as code with Terraform
  • FinOps dashboards with 4% cost reduction vs prior year
  • Warehouse sizing, auto-suspend and workload isolation
  • Role-based access models and governance rules
Data quality
  • dbt data tests across all mart models
  • Soda quality checks for operational data
  • POCs with Sifflet, Elementary and Monte Carlo
  • Benchmarked semantic layer solutions
What I took away: at scale, tools are not enough on their own. Standards, governance and automation are what turn a good tool into a good platform. dbt does not standardise how teams use it. That second part requires deliberate, sustained work.
March 2021 to February 2023
Data Engineer at Younited

This is where I genuinely grew up as an engineer. I was building and maintaining ETL jobs with Python, Spark, SQL, dbt and Airflow, often all five in the same week. I worked across Google Cloud with BigQuery, Microsoft Azure, and AWS. I set up cloud functions, optimised Spark jobs, ran POCs on Microsoft Fabric and various data flow tools, and started taking data quality monitoring seriously for the first time.

On AWS I managed infrastructure as code with Terraform and built Lambda functions to send data from Snowflake to Splunk. I also built Terraform modules for scheduling and monitoring dbt. That was my first real taste of what it looks like to treat data infrastructure as code rather than a set of manual steps you repeat every time.

I collaborated daily with data science and analysis teams to understand what the platform needed to support their work. And I kept running into the same thing: when engineers do not have a clear path, they improvise. And improvisation at scale becomes technical debt.

What I took away: infrastructure as code is the difference between a platform you can trust and one you are afraid to touch. Engineers need one clear path. Without it, you end up with fifteen variants of the same solution and nobody can maintain any of them.
March 2020 to October 2020
Data Analyst at SAP

I built real-time SAP Analytics Cloud dashboards with OData connectivity and engineered Python and SQL data pipelines. This was my first real encounter with the gap between what business teams need and what the data team actually delivers. Requirements were unclear, delivery cycles were long, and nobody agreed on what done meant for a data product.

What I took away: business alignment is a prerequisite. You cannot build good data products without understanding what decision they are supposed to support.
June 2019 to September 2019
Data Scientist at Amadeus

My first real exposure to data at scale. I was processing Logstash and Grafana pipeline data with Python, building time-series prediction models using LSTM, GRU and ARIMA. The technical work was interesting, but what stayed with me was something much simpler. The pipelines had no structure, no lineage, no tests. When something broke, you found out from a complaint, not from an alert.

What I took away: monitoring and visibility are not nice-to-haves. You need to know before your users do.
